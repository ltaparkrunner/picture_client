Для загрузки первых шести изображений лучше всего использовать двухэтапный процесс: сначала запросить список имен файлов через WebSocket (Protobuf), а затем загрузить сами данные.
Вот как это реализовать:
1. Обновите .proto схему
Добавьте сообщение для запроса списка и ответа со ссылками.
proto
message ImageListRequest {
    int32 count = 1; // Запрашиваем 6
}

message ImageInfo {
    string filename = 1;
    string url = 2;
}

message ImageListResponse {
    repeated ImageInfo images = 1;
}
Используйте код с осторожностью.
2. Реализация на сервере (Node.js)
Используйте ListObjectsV2Command для получения имен файлов из MinIO.
javascript
const { ListObjectsV2Command, GetObjectCommand } = require("@aws-sdk/client-s3");
const { getSignedUrl } = require("@aws-sdk/s3-request-presigner");

ws.on('message', async (buffer) => {
    // Если получили запрос списка (проверьте тип сообщения)
    const command = new ListObjectsV2Command({
        Bucket: 'images',
        MaxKeys: 6 // Ограничение в 6 файлов
    });

    const { Contents } = await s3Client.send(command);

    if (Contents) {
        const imageInfos = await Promise.all(Contents.map(async (file) => {
            // Генерируем временную ссылку для каждого файла
            const getCommand = new GetObjectCommand({ Bucket: 'images', Key: file.Key });
            const url = await getSignedUrl(s3Client, getCommand, { expiresIn: 3600 });
            return { filename: file.Key, url: url };
        }));

        // Сериализуем ImageListResponse и отправляем в Qt
        const responsePayload = ImageListResponse.encode({ images: imageInfos }).finish();
        ws.send(responsePayload);
    }
});
Используйте код с осторожностью.

Шаг 2: Получение списка и загрузка картинок

connect(socket, &QWebSocket::binaryMessageReceived, this, [this](const QByteArray &data) {
    ImageListResponse resp;

    if (resp.deserialize(&serializer, data)) {

        }

void downloadImage(const QString &url) {
    QNetworkAccessManager *manager = new QNetworkAccessManager(this);
    connect(manager, &QNetworkAccessManager::finished, [manager](QNetworkReply *reply) {
        if (reply->error() == QNetworkReply::NoError) {
            QPixmap pix;
            pix.loadFromData(reply->readAll());
            // Отобразить в UI (например, в QLabel или QListWidget)
        }
        reply->deleteLater();
        manager->deleteLater();
    });
    manager->get(QNetworkRequest(QUrl(url)));
}

Почему это лучший подход?
Эффективность: Вы не «забиваете» WebSocket тяжелыми бинарными данными шести картинок сразу.
Параллелизм: QNetworkAccessManager скачивает картинки параллельно (обычно до 6 соединений одновременно), что гораздо быстрее последовательной передачи через сокет.
Безопасность: Ссылки getSignedUrl временные, что защищает ваши файлы от прямого доступа извне.

Для реализации такой задачи нам нужно:
Protobuf: Создать .proto схему для упаковки картинки и метаданных.
MinIO: Загрузить бинарные данные (обычно через S3 API или просто передав массив байт на сервер, который сам положит в MinIO).
Qt: Считать файл, упаковать в Protobuf и отправить через QWebSocket.
Важный нюанс: Прямая работа с MongoDB и MinIO из C++ клиента — плохая практика (из-за безопасности). Обычно клиент отправляет Protobuf-пакет на ваш Node.js сервер, а сервер уже сохраняет данные в MinIO и MongoDB.
1. Описание Protobuf (image_message.proto)
proto
syntax = "proto3";

message ImageUpload {
    string filename = 1;
    bytes data = 2;
    string content_type = 3;
    int64 timestamp = 4;
}
Используйте код с осторожностью.
2. Функция отправки на Qt/C++
Предполагается, что вы уже сгенерировали классы из .proto файла.
cpp
#include "image_message.pb.h" // Сгенерированный файл
#include <QFile>
#include <QFileInfo>
#include <QDateTime>

void sendImage(QWebSocket *socket, const QString &filePath) {
    if (!socket || !socket->isValid()) {
        qDebug() << "Ошибка: Сокет не подключен!";
        return;
    }

    QFile file(filePath);
    if (!file.open(QIODevice::ReadOnly)) {
        qDebug() << "Ошибка: Не удалось открыть файл" << filePath;
        return;
    }

    QByteArray fileData = file.readAll();
    QFileInfo fileInfo(filePath);

    // 1. Заполняем структуру Protobuf
    ImageUpload message;
    message.set_filename(fileInfo.fileName().toStdString());
    message.set_data(fileData.constData(), fileData.size());
    message.set_content_type("image/jpeg"); // Можно определить динамически
    message.set_timestamp(QDateTime::currentMSecsSinceEpoch());

    // 2. Сериализуем в строку/массив байт
    std::string serialized = message.SerializeAsString();

    // 3. Отправляем как бинарное сообщение
    qint64 bytesSent = socket->sendBinaryMessage(QByteArray::fromStdString(serialized));

    qDebug() << "Отправлено байт:" << bytesSent << "Файл:" << fileInfo.fileName();
}
Используйте код с осторожностью.
3. Логика на стороне сервера (Node.js пример)
На сервере вам нужно будет принять этот буфер, распарсить его и отправить дальше:
javascript
const protobuf = require("protobufjs");
const Minio = require('minio');
// Подключение к MongoDB (Mongoose)

    ws.on('message', async (buffer) => {
        // 1. Декодируем Protobuf
        const root = await protobuf.load("image_message.proto");
        const ImageUpload = root.lookupType("ImageUpload");
        const message = ImageUpload.decode(buffer);

        // 2. Сохраняем в MinIO
        await minioClient.putObject('pictures', message.filename, Buffer.from(message.data));

        // 3. Сохраняем метаданные в MongoDB
        const imageMeta = new ImageModel({
            name: message.filename,
            minioPath: `pictures/${message.filename}`,
            timestamp: message.timestamp
        });
        await imageMeta.save();

        console.log(`Файл ${message.filename} успешно обработан`);
    });
});
Используйте код с осторожностью.
Почему именно так?
Бинарная передача: sendBinaryMessage эффективнее для картинок, чем Base64.
Protobuf: Гарантирует, что метаданные (имя файла) и сами данные (байты) не перемешаются и будут считаны корректно на любом языке.
Разделение ответственности: Qt клиент не должен знать пароль от MongoDB. Он просто говорит серверу: "Вот файл, сохрани его".

2. Обновленный сервер server.js
Теперь сервер принимает картинку через Protobuf, отправляет её в MinIO, а в MongoDB сохраняет только метаданные и путь.
javascript
const http = require('http');
const { Server } = require('socket.io');
const mongoose = require('mongoose');
const protobuf = require('protobufjs');
const { S3Client, PutObjectCommand } = require("@aws-sdk/client-s3");

// Настройки из окружения
const { S3_ENDPOINT, S3_ACCESS_KEY, S3_SECRET_KEY, S3_BUCKET, MONGO_URL } = process.env;

// 1. Подключение к MongoDB
mongoose.connect(MONGO_URL);
const ImageRecord = mongoose.model('ImageRecord', {
  name: String,
  s3Key: String,
  contentType: String,
  createdAt: { type: Date, default: Date.now }
});

// 2. Настройка S3 (MinIO) клиента
const s3Client = new S3Client({
  endpoint: S3_ENDPOINT,
  region: "us-east-1", // Для MinIO любое значение
  credentials: { accessKeyId: S3_ACCESS_KEY, secretAccessKey: S3_SECRET_KEY },
  forcePathStyle: true // Обязательно для MinIO
});

const server = http.createServer();
const io = new Server(server);

// 3. Загрузка Protobuf и запуск логики
protobuf.load("image.proto", (err, root) => {
  if (err) throw err;
  const Picture = root.lookupType("Picture");

  io.on('connection', (socket) => {
    socket.on('upload_image', async (buffer) => {
      try {
        // Десериализация
        const msg = Picture.decode(new Uint8Array(buffer));
        const { fileName, data, contentType } = Picture.toObject(msg);

        const s3Key = `${Date.now()}-${fileName}`;

        // А) Загрузка в MinIO
        await s3Client.send(new PutObjectCommand({
          Bucket: S3_BUCKET,
          Key: s3Key,
          Body: data,
          ContentType: contentType
        }));

        // Б) Запись в MongoDB
        const record = new ImageRecord({ name: fileName, s3Key, contentType });
        await record.save();

        socket.emit('status', { success: true, s3Key });
        console.log(`Uploaded to MinIO & Saved to DB: ${fileName}`);
      } catch (e) {
        console.error(e);
        socket.emit('status', { success: false, error: e.message });
      }
    });
  });
});

server.listen(3000, () => console.log('Server running on :3000'));

2. Команда генерации
Выполните эту команду в папке, где лежит ваш image.proto:
bash
protoc --cpp_out=. image.proto
Используйте код с осторожностью.
Результат:
image.pb.h — заголовочный файл с описанием класса Picture.
image.pb.cc — реализация методов сериализации.
3. Подключение к Qt проекту (.pro файл)
Чтобы Qt «увидел» эти файлы и смог их скомпилировать, добавьте в проект следующее:
pro
# Добавляем сгенерированные файлы
SOURCES += image.pb.cc
HEADERS += image.pb.h

# Подключаем библиотеку protobuf
# В Linux обычно так:
LIBS += -lprotobuf

# В Windows (если ставили через vcpkg, пути могут отличаться):
# INCLUDEPATH += C:/path/to/protobuf/include
# LIBS += -LC:/path/to/protobuf/lib -lprotobuf
Используйте код с осторожностью.
4. Использование в коде (C++)
Теперь вы можете использовать типизированный класс вместо сырых байтов:
cpp
#include "image.pb.h"
#include <fstream>

void ImageClient::sendImage(const QString &filePath) {
    // 1. Читаем файл в QByteArray
    QFile file(filePath);
    if (!file.open(QIODevice::ReadOnly)) return;
    QByteArray rawData = file.readAll();

    // 2. Создаем и заполняем объект Protobuf
    Picture photo;
    photo.set_filename(filePath.section('/', -1).toStdString());
    photo.set_contenttype("image/jpeg");
    photo.set_data(rawData.constData(), rawData.size());

    // 3. Сериализуем в строку/буфер
    std::string output;
    photo.SerializeToString(&output);

    // 4. Отправляем через WebSocket
    m_webSocket.sendBinaryMessage(QByteArray::fromStdString(output));
}
Используйте код с осторожностью.
Лайфхак для автоматизации в Qt
Чтобы не запускать protoc руками каждый раз, можно добавить это в .pro файл:
pro
protobuf_decl.name = protobuf header
protobuf_decl.input = PROTOS
protobuf_decl.output = ${QMAKE_FILE_BASE}.pb.h
protobuf_decl.commands = protoc --cpp_out=. ${QMAKE_FILE_NAME}
protobuf_decl.variable_out = HEADERS
QMAKE_EXTRA_COMPILERS += protobuf_decl

protobuf_impl.name = protobuf implementation
protobuf_impl.input = PROTOS
protobuf_impl.output = ${QMAKE_FILE_BASE}.pb.cc
protobuf_impl.commands = protoc --cpp_out=. ${QMAKE_FILE_NAME}
protobuf_impl.variable_out = SOURCES
QMAKE_EXTRA_COMPILERS += protobuf_impl

PROTOS = image.proto
Используйте код с осторожностью.
Теперь при каждой сборке проекта Qt сам проверит, изменился ли .proto файл, и обновит C++ код.

I work with cmake, write commands for it.
Для интеграции Protobuf в проект на CMake лучше всего использовать встроенный модуль FindProtobuf. Это позволит автоматически найти установленную библиотеку и сгенерировать файлы .pb.h / .pb.cc прямо во время сборки.
1. Настройка CMakeLists.txt
Добавьте эти строки в ваш файл:
cmake
cmake_minimum_required(VERSION 3.10)
project(QtImageClient)

# 1. Ищем Qt
find_package(Qt6 REQUIRED COMPONENTS Core Network WebSockets) # Или Qt5

# 2. Ищем Protobuf
find_package(Protobuf REQUIRED)

# 3. Указываем путь к .proto файлу и генерируем исходники
# Эта команда создаст переменные PROTO_SRCS и PROTO_HDRS
protobuf_generate_cpp(PROTO_SRCS PROTO_HDRS image.proto)

# 4. Создаем исполняемый файл
add_executable(QtImageClient
    main.cpp
    client.cpp
    client.h
    ${PROTO_SRCS} # Сгенерированные .cc файлы
    ${PROTO_HDRS} # Сгенерированные .h файлы
)

# 5. Подключаем пути к заголовочным файлам (включая папку со сгенерированными файлами)
target_include_directories(QtImageClient PRIVATE
    ${CMAKE_CURRENT_BINARY_DIR}
    ${Protobuf_INCLUDE_DIRS}
)

# 6. Линкуем библиотеки
target_link_libraries(QtImageClient PRIVATE
    Qt6::Core
    Qt6::Network
    Qt6::WebSockets
    ${Protobuf_LIBRARIES}
)
Используйте код с осторожностью.
2. Как это работает под капотом
find_package(Protobuf REQUIRED): Ищет установленный в системе protoc и библиотеки.
protobuf_generate_cpp: Добавляет в процесс сборки вызов protoc. Файлы будут созданы в папке сборки (build), чтобы не «мусорить» в папке с исходным кодом.
${CMAKE_CURRENT_BINARY_DIR}: Мы добавляем эту папку в include_directories, чтобы ваш код мог сделать #include "image.pb.h", не зная точного пути.
3. Совет по компиляции
Если вы используете vcpkg, не забудьте передать путь к нему при первой конфигурации проекта:
bash
cmake -B build -S . -DCMAKE_TOOLCHAIN_FILE=[путь_к_vcpkg]/scripts/buildsystems/vcpkg.cmake
cmake --build build
Используйте код с осторожностью.
4. Важный нюанс для Windows (DLL)
Если вы собираете проект на Windows и Protobuf установлен как динамическая библиотека, добавьте это перед поиском пакета, чтобы избежать ошибок линковки:
cmake
add_definitions(-DPROTOBUF_USE_DLLS)
